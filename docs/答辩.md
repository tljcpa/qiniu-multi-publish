# 答辩脚本：多平台内容发布工具

> 本脚本面向 5 分钟现场答辩，配合在线 demo [https://publish.qiniu.zdwktlj.top](https://publish.qiniu.zdwktlj.top) 使用。
> 评委可全程跟着脚本在浏览器里复现每一步。脚本里的台词是可以照着念的，不是大纲。

---

## 一、5 分钟时间分配脚本（逐段台词）

整体节奏：前 45 秒先抛出一个反直觉的产品决策抓住注意力，中间用架构和效果证明这个决策是认真做出来的，最后一分钟现场跑一遍闭环收尾。

### 0:00 - 0:45 产品决策：我们故意不做"真发布"（WOW 时刻）

> "题目要求一份内容自动适配多个平台、支持一键发布。我们做了一个看起来反直觉的决定：**我们不做真发布。**
>
> 原因很简单——公众号、知乎、B站、小红书都没有开放的第三方发布 API。市面上号称'一键全平台发布'的工具，要么是逆向接口（随时封号），要么干脆是骗你。**假装能发，本身就是一种欺骗。**
>
> 所以我们重新定义了'一键发布'：把它拆成'内容适配 → 平台预览 → 一键复制到剪贴板 → 跳转到平台官方编辑页'。真正按下'发布'那一下，留在用户手里、留在平台官方界面里。
>
> 这样做的价值不是少了，而是更实在：把一篇文章发到 4 个平台，从大约 30 分钟手工改写排版，压缩到大约 3 分钟。我们不解决一个假问题，我们把真问题解决得很顺手。"

要点：这一段不要露怯，要主动把"不做真发布"讲成优势而不是缺陷。把节省时间的量化数字（30 分钟 → 3 分钟）说出来。

### 0:45 - 1:45 插件化架构：加一个平台只要约 25 行 + 一行注册

> "题目还要求给出'扩展更多平台的架构设计'。我们的答案是 Platform Adapter 插件化架构。
>
> 所有编排逻辑——组装 prompt、调 LLM、解析 JSON、渲染、生成跳转意图——全部写在抽象基类 `PlatformAdapter` 里。新增一个平台时，你不写任何编排代码，只回答两个问题：**这个平台是谁，它的内容长什么样。**
>
> 具体到代码：你新建一个 adapter 类，填四个必填项——平台标识 `name`、中文名 `display_name`、官方编辑页 `editor_url`、还有灵魂所在的 `style_prompt()`，然后加一个 `@register` 装饰器。就这些，**约 25 行，没有一行 HTTP 或编排代码。** 我们扩展指南里那个虚构的'示例号'平台，整个实现就是 25 行。
>
> 注册之后，`/platforms` 接口和前端预览会自动发现这个新平台，核心代码零改动。这就是题目所说'扩展架构'的工程兑现。"

要点：如果有投屏，这里切到 EXTENSION_GUIDE 第 3 节的 ExamplePlatform 代码块，手指点着 25 行那段念。强调"零编排代码""核心代码零改动"。

### 1:45 - 3:00 风格适配深度：我们做的是风格，不只是格式

> "很多人以为多平台适配就是 Markdown 转 HTML，那只是格式。**真正难的是风格。** 同一篇内容，在不同平台得是不同的语气、不同的标题习惯、不同的段落结构、不同的互动话术。
>
> 给大家看同一篇《远程办公真的更高效吗》原文，在 4 个平台被改成了什么标题：
>
> - 公众号：**远程办公效率低？关键在这两点** —— 设问 + 给确定性，标题克制在 22 字内
> - 知乎：**远程办公真的更高效吗？关键在异步协作与文档文化** —— 专业、信息量大、像一个认真的回答
> - B站：**远程办公真的更高效？我愿称之为"血赚"还是"血亏"** —— 玩梗、口语、有网感
> - 小红书：**❌别再被忽悠了！远程办公的真相是……** —— emoji 开头、制造悬念、强情绪
>
> 不只是标题。正文也跟着变：公众号 emoji 克制到 3 个以内、知乎几乎不用 emoji、小红书 emoji 是核心风格；段落长度、结尾的互动话术（三连 / 评论区扣 1 / 收藏）、话题标签密度，全都按平台调。这些规则都写在每个平台 adapter 的 `style_prompt()` 里，是可执行的具体规则，不是泛泛而谈。"

要点：4 个标题是全场最有说服力的物料，念慢一点，让评委体会到差异。小红书标题里的 ❌ 是平台内容本身的风格，照念即可。

### 3:00 - 4:00 高保真所见即所得预览：4 个手机壳并排

> "适配出来的内容长什么样，我们不让用户脑补。前端为每个平台做了高保真的'手机壳'预览：用 CSS 仿真各平台的真实 UI——公众号是文章页、知乎是回答卡片、B站是专栏页、小红书是笔记卡片。
>
> 4 个手机壳并排放，用户一眼就能看到同一篇内容在 4 个平台分别是什么效果，所见即所得。这一步解决的是信任问题：你不是把内容交给一个黑盒，你看得见结果再决定。"

要点：投屏切到 demo 的预览面板，让 4 个手机壳同屏出现。如果现场网络好就用线上 demo，网络差就用预先截好的图（见截图清单）。

### 4:00 - 5:00 现场 demo：跑通完整闭环

> "最后现场跑一遍。我粘贴一篇博客原文进来，点'一键适配'……（等待约 6.5 秒，4 平台并发适配完成）……4 个平台的手机壳预览同时出来了。
>
> 我点知乎这张卡片下面的'复制并前往知乎发布'——内容已经复制到剪贴板，同时新标签页打开了知乎官方的写文章页面。我在知乎官方界面里粘贴，确认无误，由我自己按下发布。
>
> 整个闭环：一份原文，4 平台风格化，复制 + 跳转，发布权始终在用户手里。这就是我们的多平台内容发布工具。谢谢。"

要点：留足 demo 的容错时间。适配大约 6.5 秒，提前告诉评委"它在并发调 4 个平台，稍等几秒"，避免冷场。

## 二、现场 demo 具体脚本

### 用哪篇内容

固定用《远程办公真的更高效吗》这篇原文（与上面 4 个标题对应，效果可预期、可对照）。提前把原文文本存在本地一个 txt 里，现场直接粘贴，不要现场打字。

### 操作步骤（点哪些按钮、预期看到什么）

1. 打开 [https://publish.qiniu.zdwktlj.top](https://publish.qiniu.zdwktlj.top)，确认页面已加载、编辑器可见。
2. 把《远程办公真的更高效吗》原文粘进 Tiptap 编辑器。预期：编辑器正常显示 Markdown 渲染。
3. 勾选目标平台（公众号 / 知乎 / B站 / 小红书 4 个全选），点"一键适配"。预期：约 6.5 秒后，4 平台并发适配完成。
4. 看 4 个平台手机壳并排预览。预期：4 个标题明显不同（与脚本第一部分列的 4 个标题对应），正文风格各异。
5. 在知乎卡片下点"复制并前往知乎发布"。预期：提示"已复制"，浏览器新标签页打开知乎官方写文章页 `editor_url`。
6. （可选，时间够才做）切到新标签，在知乎官方编辑器里 Ctrl+V 粘贴，展示内容已就位。强调：发布按钮是知乎自己的，由我按下。

### 容错与兜底话术

- **某平台适配慢**：提前铺垫"它在并发调用 4 个平台的 LLM，正常 6 秒左右"。超过 10 秒就一边等一边讲架构，不要盯着进度条沉默。
- **某平台适配失败**：系统是逐平台独立适配的，**单平台失败不拖垮整体**，其余 3 个照常出结果。话术："注意这里——某个平台超时了，但另外三个完全正常。这是我们刻意的容错设计，各平台 adapter 互相独立，一个挂了不影响其它。"这反而是个加分点，顺势讲架构解耦。
- **现场网络差 / demo 站打不开**：直接切到预先截好的全套截图（见下方截图清单），按同样顺序讲一遍。截图必须提前备齐，这是 demo 的保险。
- **剪贴板复制无效**：剪贴板 API 要求 HTTPS，demo 站本身是 HTTPS 没问题；若浏览器弹权限请求，提前点允许。

## 四、预案问答

### Q1：这不就是 ChatGPT 套壳改写吗？

套壳改写解决的是"改一段文字"，我们解决的是"一份内容发到 4 个平台的完整工作流"。三点不一样：一，我们把平台差异沉淀成了**可插拔的 Adapter 架构**，每个平台的风格规则是工程资产，不是一句临时 prompt；二，我们做了**高保真的所见即所得预览**，用户看得见每个平台的真实效果再决定，这是裸 ChatGPT 给不了的；三，我们做了**复制 + 跳转的闭环**，把内容直接送进平台官方编辑器。LLM 只是其中一个被抽象掉的组件，换成别的模型对架构无感。

### Q2：平台 API 不开放，你这工具商业价值在哪？

商业价值恰恰来自 API 不开放。正因为没有官方多平台发布通道，内容创作者和运营才不得不手工一篇一篇改写排版，这是真实存在的、每天都在发生的痛点。我们把它从约 30 分钟压到约 3 分钟，对一个日更多平台的运营来说就是每天省下半小时。我们不赌"哪天 API 开放"，我们解决"API 永远不开放"这个现实下的效率问题。

### Q3：风格适配会不会丢失作者本人风格？

会有这个风险，所以我们的定位是**辅助而非替代**。风格适配做的是"把你的内容翻译成平台听得懂的语气"，最终内容在跳转后由作者在官方界面终审，作者可以改、可以删、可以不用。而且我们保留了原始草稿和各平台版本（落在 SQLite 里），作者随时能对照原文。风格是平台维度的本地化，不是抹掉作者，是帮作者在每个平台说"当地话"。

### Q4：怎么保证 LLM 不乱改、不编造数据？

三道防线。第一，所有平台的 `style_prompt()` 末尾都强制写了"保持原文事实，不要编造数据"——这是我们踩坑后加的，曾经小红书版本自己补了原文没有的工具名，发现后统一加固。第二，LLM 用 JSON mode + Pydantic schema 校验，结构不合法就重试，脏数据进不了前端。第三也是最关键的，**最后由用户在平台官方界面终审**，我们从不替用户按发布。我们诚实承认 LLM 仍可能有轻微创意漂移，所以把终审权留在人手里，这正是"不做真发布"决策的另一重价值。

### Q5：扩展新平台到底多快？用架构事实说。

约 25 到 100 行代码、5 分钟。具体成本是：在 `backend/adapters/` 新建一个文件，写一个继承 `PlatformAdapter` 的类，填四个必填项（`name` / `display_name` / `editor_url` / `style_prompt()`），加一个 `@register` 装饰器，在 `__init__.py` 加一行 import。其余方法——few-shot、输出 schema、格式化、预览模板、跳转意图——全有可复用默认实现，按需覆写。核心代码（路由层、注册表、前端预览框架）零改动，`/platforms` 接口和前端预览自动发现新平台。我们还有统一契约测试 `run_contract`，任何新 adapter 一跑就知道是否满足系统假设。

### Q6：为什么不做真发布 / 自动发布？

因为做不到诚实的真发布。四个平台都没有开放的第三方发布 API，要"自动发布"只能逆向接口，结果是账号随时被封、随时失效——这是给用户埋雷，不是帮用户。我们选择把"按下发布"这一步明确交还给用户、交还给平台官方界面。这不是能力不足的妥协，是产品伦理上的主动选择：我们对能做和不能做的边界保持诚实。

### Q7：四个平台并发适配，性能怎么样？一个慢的会不会拖垮整体？

实测 4 平台并发适配约 6.5 秒。后端用 `asyncio.gather` 并发调用各平台 adapter，所以总耗时接近单个平台耗时，不是 4 倍累加。各平台 adapter 完全独立、互不依赖，**单平台失败或超时不会拖垮其它平台**，其余平台照常返回结果。工程上我们也有保障：31 个离线契约测试（用 FakeProvider，CI 无需 LLM key 即可全绿）+ 4 个平台各自的 DeepSeek live 测试，前端构建产物 221KB gzip。

### Q8：为什么前端要做"手机壳"这么重的预览，直接出文本不行吗？

因为它解决的是**信任**问题。用户最担心的是"AI 改完到底长啥样、发出去会不会丑"。纯文本预览看不出排版效果，用户还是得去平台试。我们用 CSS 仿真每个平台的真实 UI，4 个手机壳并排，所见即所得——用户在点跳转之前就确信结果对了。这一步把"内容黑盒 + 跳转后才发现问题"的体验，变成了"先看准再行动"，是闭环里降低决策成本的关键一环。
