### D-01: 产品定位（真发布 vs 模拟发布 vs 适配+复制+跳转）
**本项目灵魂决策。** 时间：2026-05-29。
- 选项 A：浏览器自动化 / RPA 真发布
- 选项 B：模拟发布 UI（点"发布"按钮但什么都没发生）
- 选项 C：诚实定位为"适配 → 预览 → 复制到剪贴板 → 跳转平台官方编辑页"
- **选 C。** 为什么：A 法律灰色 + 反爬对抗远超 3 天；B 是欺骗用户，评委一眼看穿"半成品"；C 把"按下发布"那一步永远留在用户手里、平台官方界面里，对创作者隐私和平台规则诚实。
- 加分逻辑：诚实定位 + 优秀适配/预览 = 评委看到"认真做产品的人"，省下时间投入架构设计（题目原文必交付物）。
- 改变条件：未来平台开放官方内容发布 API（短期不会发生）。

### D-02: 后端框架（FastAPI vs Express vs Django）
时间：2026-05-29（PR1 定）。
- 选 **FastAPI（Python 3.11）**。
- 为什么：async 原生适合 LLM 流式；Pydantic 给 LLM JSON schema 输出做强类型校验；openai-python 直连 DeepSeek（OpenAI 兼容协议）。
- 为什么不 Django：太重，无复杂 ORM/admin 需求，SQLite 足够。
- 为什么不 Express：默认 Python 栈，LLM 生态在 Python 更顺。
- 改变条件：若需 SSR 一体化，再考虑 Next.js 全栈。

### D-02b: 仓库布局（monorepo: backend/ + frontend/）
时间：2026-05-29（PR1 定）。
- 选 **monorepo**：`backend/`（FastAPI + adapters 包）+ `frontend/`（React+Vite）+ `docs/` + 根 `docker-compose.yml`。
- 为什么：单人 3 天项目，前后端强耦合演进，monorepo 一次 clone、一个 compose 起全栈、PR 跨层方便。
- `adapters/` 放在 `backend/` 下作为独立 Python 包，对应 README 里 `from adapters.base import ...` 的扩展示例。
- 改变条件：若平台 adapter 要做成可单独 pip 分发的插件包，再拆仓。

### D-03: 富文本编辑器（Tiptap vs Quill vs Slate vs Monaco）（PR11）
时间：2026-05-29。
- 选 **Tiptap**（@tiptap/react + starter-kit + tiptap-markdown）。
- 为什么：基于 ProseMirror，现代、可控、扩展性强；`tiptap-markdown` 能直接把编辑内容序列化成 Markdown，正好喂后端 `body_md`，避免 HTML→MD 转换坑。
- 为什么不 Quill：API 偏老、定制样式麻烦；不选 Monaco：那是代码编辑器，不适合图文创作。
- 改变条件：若要协同编辑再考虑 Yjs + Tiptap collaboration。

### D-03b: 前端技术细节（PR11）
时间：2026-05-29。
- Vite + React + TypeScript + **Tailwind v3**（不用 v4，配置更稳）。
- **暂不引入 shadcn/ui**：其 CLI 生成器在手写+VM 构建流程里增加摩擦，先用 Tailwind 手写组件 + lucide-react 图标，够好看且零生成步骤；后续有空再补。

### D-04: PlatformAdapter 接口抽象的最小集（PR4）
时间：2026-05-29。核心问题：哪些方法**必须**实现、哪些给默认值。
- **必须（abstract）**：`name` / `display_name` / `editor_url` / `style_prompt()`。
  理由：一个新平台之间真正的差异只有"身份 + 风格"。少了任一个就无法适配或跳转。
- **可选（有默认实现）**：`few_shot()`=[]、`output_schema()`=通用 {title,content,summary,hashtags}、
  `format_content()`=按 schema 拼通用 HTML、`preview_template()`=通用卡片、
  `publish_intent()`=用 editor_url + format_content 组装 {clipboard,url}、`extension_guide()`=""。
- **模板方法 `adapt(content, provider)` 放基类做成具体方法**：组装 system(style_prompt)+few_shot+user，
  调 provider.chat_json。子类只改"风格描述/样本/schema"，不碰编排逻辑——这就是"约 100 行一个平台"的来源。
- 为什么这样切：把"每个平台必然不同的"设为 abstract，"大多数平台可复用的"给默认，最大化新增平台的省力度。
- 为什么不把 format/preview 也设为必填：会让加平台成本翻倍，违背"5 行注册"目标；需要时再 override。
- 改变条件：若某类平台（如视频平台）连内容模型都不同，再引入第二层抽象基类。

### D-05: 风格适配 prompt 工程（PR5）
时间：2026-05-29。
- 选 **详细 zero-shot 风格描述 + JSON schema 约束 + temperature=0.7** 为主线，few-shot 作为可选增强（默认不开）。
- 为什么：① 详尽的风格规则（语气/段落/标题/互动话术）本身就能稳定区分四平台；② JSON mode + schema 保证结构可解析；③ temperature=0.7 平衡稳定与创意（太低呆板、太高跑题）。
- 为什么不强依赖 few-shot：余额仅 ¥11，few-shot 抬 token；且真实平台内容有版权风险（见 D-06）。
- 改变条件：若某平台 zero-shot 风格不够像（如小红书 emoji 密度），再补**自己原创**的风格示范片段做 few-shot。

### D-08: 跨平台剪贴板兼容（PR13）
时间：2026-05-29。

### D-16: 前端设计语言——去 AI 味、做"领域工具"（PR21）
时间：2026-05-30。wly 把"反 AI 味"列为前端硬验收标准。
- **明确禁止**：紫→粉/靛蓝渐变、毛玻璃、霓虹光晕、标题/功能位 emoji、shadcn 默认 slate/zinc、营销三连词、占位文案、大留白三列特性网格。

### D-15: 流式适配打字机效果（PR19）
时间：2026-05-29。
- `POST /adapt/stream` 用 SSE，多平台并发"打字机"：每平台一个线程跑 provider.chat_stream，经 asyncio.Queue 汇流到一个 SSE 响应，4 个手机壳同时逐字显示。
- 难点：结构化输出(JSON mode)不利于流式。取舍——流式端点改用**纯文本约定格式**（首行标题 / 正文 / 末行 TAGS:），边流边显原始文本，结束后本地 parse 成结构化 title/content/hashtags。一次 LLM 调用既给 typewriter 又给结构化结果。
- 为什么不流式 JSON：增量 JSON 不可解析、体验差；纯文本格式对模型更自然、解析容错好。
- Caddy 需放行 /adapt/stream 且 reverse_proxy flush_interval -1 关缓冲，否则 SSE 被攒着一次性吐。
- 改变条件：若要严格结构化流式，可上 OpenAI 的 streaming + 分字段协议（成本高，demo 不值）。

### D-11: 部署方案 + HTTPS（PR14）
时间：2026-05-29。
- 后端 **Docker Compose**（容器 8082）；前端 **VM 上 node 构建静态 dist**，由 **系统 Caddy** 静态托管 + 按路径反代 API；HTTPS 走 Caddy 自动 Let's Encrypt。
- 三项目共机隔离用 **Caddy conf.d drop-in**：主 Caddyfile `import /etc/caddy/conf.d/*.caddy`，本项目独占 `multi-publish.caddy`（子域名 publish.qiniu.zdwktlj.top），互不踩。
- 为什么前端不进 docker：host 构建 + Caddy 直接服 dist 更省内存（VM 仅 3.8G）、改前端只需 rebuild dist 不重启容器，demo 迭代快。
- 为什么必须 HTTPS：navigator.clipboard 仅安全上下文可用（剪贴板是核心闭环），Caddy 自动证书一步到位。
- 改变条件：若要一键迁移环境，可把前端也做成 nginx 静态容器纳入 compose。

### D-12: LLM Provider 抽象设计（PR3）
时间：2026-05-29。
- 关键：DeepSeek 与 Azure OpenAI **都兼容 OpenAI SDK**，抽出 `_OpenAICompatProvider` 共享 chat/chat_stream/chat_json，子类只负责构造 client。
- 提供三个能力：`chat`(纯文本) / `chat_stream`(流式，给前端打字机效果) / `chat_json`(JSON mode + 解析失败重试)。
- 为什么 chat_json 用 `response_format=json_object` 而非自己 parse 文本：DeepSeek/OpenAI 原生 JSON mode 更稳；注意 DeepSeek 要求 prompt 里出现 "json" 字样，否则报错（已在实现里强制）。
- 为什么不引入 LangChain：套壳印象 + 不可控（见 BRIEF §5.2 明确不用）。
- 改变条件：若要接非 OpenAI 协议后端（如本地 vLLM 非兼容模式），再加独立子类。
- 测试策略：单测用 fake client 不打网络；真连 DeepSeek 的 smoke 测试在无 key 时自动 skip，保证 main 始终绿。

### D-13: 后端 /adapt 的并发与容错（PR10）
时间：2026-05-29。
- 多平台适配用 `asyncio.gather` 并发，每个 adapter 的同步阻塞调用丢进 `asyncio.to_thread`。
- 为什么：4 个平台串行 ≈ 4×单平台耗时(~10s)，并发后 ≈ 最慢单平台(~3-5s)，直接决定 demo 流畅度。
- 容错：**部分失败不拖垮整体**——单平台异常时该平台 result.error 非空，其余照常返回。demo 时某平台抽风也不白屏。
- provider 实例只构造一次复用，避免每平台重建 OpenAI client。
- 改变条件：若要做流式打字机效果，再加 `/adapt/stream`（SSE），当前先保证整体可用。

### 5.4 AI 出过的错与修正
- **风格适配的创意漂移**：小红书适配'远程办公'时，LLM 自行补了"推荐飞书/Notion"等原文没有的工具名。原因：种草风格 prompt 鼓励"安利"，模型顺势编造。已在所有 style_prompt 末尾强制"保持原文事实，不要编造数据"，并在 EXTENSION_GUIDE 第 5 节列为写 prompt 的必备经验；演示时也会主动说明这是 LLM 风格化的已知边界，由用户在跳转后的官方界面终审。
- **DeepSeek JSON mode 的隐性要求**：首版 chat_json 直接传 response_format=json_object 报 400，原因是 DeepSeek 要求 prompt 里出现 "json" 字样。已在 chat_json 里自动补一条系统提示兜底（见 L 待补 / 复盘 D-12）。
- **base.py 残留未用 import**：PR4 写 base 时引入了 `import html` 却没用到，code review 时删除。教训：写完即自检 import。
